功能测试

分类

按是否覆盖源代码

黑盒：不关注代码逻辑，只关注输入输出。

白盒：关注代码逻辑。

灰盒测试：多用于集成测试阶段。

按测试阶段

单元测试：单个功能模块，对软件中最小可测单元进行检查和验证

集成测试：组装测试，组装成完整功能，将所有程序模块进行有序的、递增的测试，重点测试不同模块的接口部分。

系统测试：将整个软件系统看成一个整体进行测试，依据是软件需求说明书，软硬件

验收测试：检验软件是否符合用户需求的测试：α：内测 β：公测 γ：候选发布版本

按是否运行

静态：不运行，界面或文档可能存在

动态：实际运行被测程序

按是否自动化

手工测试

自动化测试

更多

冒烟测试：针对最基本功能，保证基本的功能和流程能走通。

回归测试：修复一个bug后，把之前的测试用例在新的代码下再次测试。

随机测试：每个人不同的经验、角度。

探索测试：

软件测试模型

瀑布模型

职责明确，不响应需求的变化。应用于大型项目。

可研与计划－需求分析－概要设计－详细设计－编码－软件测试－运行维护

可研与计划：可行性研究报告

需求分析：需求规格说明书

概要设计：概要设计文档（架构师）

详细设计：详细设计文档（开发）

编码：敲代码

软件测试：测试报告

V模型

需求分析-概要设计-详细设计-编码-单元测试-集成测试-系统测试-验收测试

测试的活动更多，并没有解决瀑布模型不响应需求变化的缺点。

W模型

  • 开发V：需求分析-概要设计-详细设计-编码-集成-实施-交付
  • 缺点：技术和管理要求比较高
  • 测试V：验收系统测试设计-集成测试设计-单元测试设计-单元测试-集成测试-系统测试-验收测试
  • 测试参与了软件开发全生命周期

软件质量模型

功能性：检察业务功能是否满足需求

可靠性：容错性能、纠错能力

易用性：易读易懂

效率：性能（响应时间、消耗的资源）

可维护性：为后续功能的开发与维护提供便利

可移植性：不同的软件环境和硬件环境

测试用例

概念

一个为了特定目的（验证产品的功能实现是否满足用户需求）而设计的的包含（测试输入、执行条件、预期结果）的文档。

好处

  • 便于理清测试思路，确保需覆盖测试的功能点无遗漏。
  • 便于测试工作量的评估
  • 便于提前准备测试数据
  • 便于把控测试工作进度
  • 便于回归测试
  • 便于测试工作的组织，提高测试效率，降低测试交接成本

组成要素

  • 用例编号
  • 用例标题
  • 测试项目
  • 用例级别
  • 预置条件
  • 测试输入
  • 执行步骤
  • 预期结果

等价类划分法

概念

在所有测试数据中，具有某种共同特征的数据子集。

划分

满足要求：有效等价类

不满足要求：无效等价类

应用场景

有大量数据测试输入，但是没法穷举测试。如输入框、下拉列表、单选复选框，需要同时提交，对于每种输入都需要大量测试输入验证

典型代表：页面级的输入框类测试

步骤

  1. 需求分析

  2. 划分等价类：

    • 有效：满足所有需求

    • 无效：规则（需求）、长度、类型、是否为空（必填项）、是否重复。只要一项不满足即可。

  3. 设计测试用例

实战

一、QQ账号：

需求分析：6-10位自然数

长度：6-10位

类型：自然数

划分等价类：

有效等价类：6-10位自然数，有效数据：123456

无效等价类：

长度：小于5位、大于10位

类型：非自然数，包含字母、符号、中文、空格，空

二、电话号码：

需求分析：验证某城市电话号码正确性

区号：空或三位数字

前缀码：非0且非1开头的三位数字

后缀码：四位数字

有效等价类：

无效等价类：

三、新浪邮箱登录

需求分析：

测试目的：邮箱能否登录

测试条件：

邮箱名：4-16位字符，支持英文、数字、下划线（不能全是数字或者下划线）

密码：6-18位字符

有效等价类：

无效等价类：

挑一个最小的一个最大的

注意事项

单模块测试中，用例标题具有唯一性。

必要步骤尽可能清楚。

边界值分析法

场景

开发人员常常在边界的位置容易出现问题，因此需要针对边界位置再测试。

对于等价类划分法的完善和补充。

针对有边界范围的批量数据的输入类测试（重点关注边界）

典型代表：输入框（有边界范围区间）

边界范围确定

选取正好等于、刚好大于、刚好小于边界的值作为测试数据。

  • 上点：边界上的点（正好等于）
  • 离点：距离上点最近的点（刚好大于、刚好小于）
  • 内点：范围内的点（区间范围内的点）

步骤

明确需求：测试目的、测试条件

划分等价类：有效等价类、无效等价类

确认边界范围值：上点、离点（可以优化）、内点（可以和有效等价类的取值合并）

提取数据编写用例：

上点需要单独列出。

边界取值优化

内点：必选（建议选择中间范围）

上点：必须选

离点：可以优化，取和上点取值相反的取值点即可（上点有效，离点选无效 ）

案例

一、QQ

上点：6 位、10位

离点：5位、11位

内点：8位

二、标题长度

标题长度大于0，小于等于30个字符。

边界值：

上点：0个、30个

离点：1个，31个

内点：15个

三、邮箱登录

邮箱名边界值：

上点：4，16

离点：3，17

内点：10

密码边界值：

上点：6，18

离点：5，19

内点：12

判定表法

案例

若用户欠费或者关机，则不允许主被叫

适用场景

有多个输入条件，多个输出结果，输入条件之间有组合关系，输入条件和输出结果之间有依赖（制约）关系。

常见词汇：如果…那么…

局限性：判定表一般适用于条件组合数量较少的情况。

判定表构成

条件桩：列出问题中所有条件。

动作桩：列出问题中可能采取的操作。

条件项：列出条件的对应取值。

动作项：列出条件项的各种取值情况下应该采取的动作结果。

规则：判定表中贯穿条件项和动作项的一列就是一条规则。

步骤

明确需求

画出判定表：

  • 列出条件桩和动作桩
  • 填写条件项，对条件进行全组合
  • 根据条件项的组合确定动作项
  • 简化、合并相似规则（有相同的动作）

根据规则编写测试用例

案例

一、订购单检查

二、文件修改

如想对文件进行修改，需要遵守以下规则： 输入的第一列字符必须是A或B，第二列字符必须是一个数字， 如果第一列字符不正确，则给出信息L； 如果第二列字符不正确，则给出信息M。 如果两列字符输入正确，则修改文件成功， 如果两列字符都输入错误，则给出信息L、M。

正交法

场景法

也叫流程图法，是用流程图描述用户的使用场景，然后通过覆盖路程路径来设计测试用例。

意义

用户使用角度：用户平时使用的不是单个功能，而是多个功能组合起来进行使用。

测试人员角度：平时测试的都是单个功能点，容易忽略多个功能的组合测试。

适用场景

根据实际的应用场景，对于多个功能之间的组合测试，可以使用场景法。

一般在测试的后期，对整个系统的模块功能进行全部的组合测试。

步骤：

  • 需求分析
  • 绘制流程图
  • 设计测试用例

错误推测法

要求：有实际项目测试经验的人使用。

定义：通过直觉（经验）或者智慧推测系统可能出现问题的地方进行再次测试。

适用场景

1.时间紧迫：通过以往类似项目的经验，提取当前项目中核心模块（出现问题较多）进行再次测试。

2.时间宽裕：在基础测试的基础上，将原有模块（存在问题较多）进行再次细化测试。

状态迁移法

是一种基于产品规格分析，对系统的每个状态及与状态相关的功能进行测试，通过不同的状态验证程序的逻辑流程。

适用场景：

  • 在业务流程中涉及到了复杂的业务场景（即业务状态的迁移）。
  • 这些业务场景在需求规格说明书中往往不能够完全阐述清楚，容易出现遗漏。

使用步骤

  1. 明确状态节点：分析被测试对象的需求规格说明，明确被测对象的状态节点数量。
  2. 绘制状态迁移图：利用圆圈表示状态节点，有向箭头表示状态间的迁移关系。
  3. 绘制状态迁移树：根据状态迁移图的节点和箭头绘制状态迁移树
  4. 抽取测试路径设计用例：根据迁移树抽取测试路径，从左至右，横向抽取，每条路径构成一条测试规则。

案例分析1–飞机售票系统

  1. 明确状态节点：预定、已支付、已出票、已使用、已取消
  2. 绘制状态迁移图
  3. 绘制状态迁移树
  4. 设计用例

业务流程测试

流程图

  • 椭圆：开始/结束
  • 箭头：路径、流程的走向
  • 平行四边形：数据的输入/输出
  • 矩形：处理/步骤/过程
  • 菱形：判断

绘制原则：

  • 流程正确，不能遗漏路径
  • 先判定，再有判断结果
  • 主流程保持在流程图的中间位置，方便阅读

业务流程测试

  • 关注点在核心业务是否能够跑通
  • 重点不是关注单个功能模块的细节点

测试时机：

  • 上线前进行业务流程测试的确认
  • 单功能模块基本可用的情况下，尽早进行（冒烟测试）

用例设计

  • 需求分析，明确流程
  • 画出流程图
  • 编写测试用例

软件缺陷

定义：软件在使用过程中存在的任何问题，都叫软件的缺陷，简称bug。（依据：需求说明书）

判定标准

  • 软件未实现需求说明书中明确要求的功能–中高级（没有做）

  • 软件出现了需求说明书中指明不应该出现的错误–中高级（做错了）

  • 软件实现的功能超出需求说明书指明的范围（做多了）

  • 软件未实现需求说明书中虽未明确指明但应该实现的要求（没做好）

  • 软件难以理解，不易使用，运行缓慢，用户体验不好。（不完美）（优先级较低）

产生原因

  • 需求阶段：需求描述不易理解，有歧义、错误等。
  • 设计阶段：设计文档存在错误或者缺陷。
  • 编码阶段：代码出现错误。
  • 运行系统：软硬件系统本身故障导致软件缺陷。

软件缺陷核心内容

  • 缺陷的标题——描述缺陷的核心问题
  • 缺陷的预置条件——缺陷产生的前体
  • 缺陷的复现步骤——复现缺陷的过程
  • 缺陷的预期结果——希望得到的结果
  • 缺陷的实际结果——实际得到的结果
  • 缺陷的必要附件——图片、日志等信息

缺陷报告的其他要素

  • 缺陷的编号”能够唯一的表示一个缺陷
  • 缺陷的状态：描述缺陷生命周期的过程
    • 新建（new）
    • 打开（open）：表示开发确认通过
    • 拒绝（rejected）：表示开发确认不通过
    • 进行中（inprogress）：表示开发正在修复缺陷
    • 已修复（fixed）：回归测试
    • 延迟修复（delay/postpone）
    • 测试通过（closed）
    • 测试不通过（reopen/open）
  • 缺陷的所属模块：描述缺陷产生的模块范围
  • 优先级：依据：1.测试用例优先级2.产品经理定的标准
  • 严重级
  • 缺陷的类型：
    • 功能问题
    • UI
    • 兼容性
    • 易用性
    • 架构
    • 安全性

编写缺陷报告规范

  • 可复现
  • 唯一性
  • 规范性：遵循公司规定的报bug的要求

注意事项：

  • 确保上报的bug是准确的

面试题

在实际测试中出现不可复现的bug怎么办？

1.经过多次复现后，还是没有出现，此时在本地记录当前问题
2.回顾当时操作的流程及测试环境的配置要求，确认是否是由于操作失误或环境临时故障引起
3.请开发协助查找当前测试模块是否有对应的日志信息
4.考虑更换一套环境查看是否能够复现上述问题
5.在后续版本测试中，需要关注当时测试该功能时是否还出现上述的问题
6.在后续版本还出现过，需要开发协助打印日志进行分析定位，同时测试需要提交bug

使用流程

管理用例

  • 创建用例：将需求文档直接转换为可测试的功能点的过程
  • 评审用例：查看已经编写好的用例是否有错误、遗漏、不正确的地方 进行评估的过程
  • 执行用例

管理缺陷

  • 缺陷的创建
  • 缺陷的跟踪
  • 缺陷的验证

非功能测试

基本测试策略：

  • 冒烟测试：基本功能检查、核心业务流程测试
  • 单功能测试
  • 集成测试与回归
  • 系统测试与回归
  • 验收测试与回归

兼容性测试

概念：不同平台、系统都能正常工作

测试关注点（Web）：

  • 浏览器：IE、Chrome、FireFox
  • 操作系统：Windows、Linux、mac等
  • 网络
  • 分辨率

界面测试

测试用户界面的功能模块的布局是否合理、整体风格是否一致、各个控件是否符合客户使用习惯

  • 布局
  • 风格
  • 按钮
  • 测试基本原则：参照UI设计图
  • 若无原型图：
    • 导航测试
    • 图形测试
    • 内容测试（有无错别字）
    • 整体界面风格测试

易用性

易懂、易学

关注点：

  • 用户群体
  • 计算机水平
  • 项目复杂性

性能测试

什么时候考虑：

  • 对性能有要求
  • 用户量大

目的：

验证稳定性、验证是否达到性能目标

并发测试、压力测试、负载测试

安全性

关注点：

输入数据：登录时密码是否进行加密以及密码是否容易破解，敏感信息遮挡处理，不能复制

传输数据：加密要有复杂度

输出数据：数据库中敏感信息要加密
